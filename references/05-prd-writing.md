# PRD 写作

## PRD 的目标

PRD 不是堆功能，而是让设计、研发、测试、运营理解：为什么做、做什么、边界是什么、怎样算完成。

## PRD 类型

- **轻量 PRD**：用于早期讨论和小功能。
- **标准 PRD**：用于进入研发设计协作。
- **技术评审版 PRD**：更强调规则、状态、异常、权限、数据和验收标准。
- **汇报版方案**：更强调背景、价值、取舍和路线图。

## 用户故事写法

格式：
> 作为【用户角色】，当【场景/触发条件】时，我希望【能力/行为】，以便【用户价值/业务结果】。

好的用户故事要有：
- 明确用户角色
- 明确触发场景
- 明确期望能力
- 明确价值结果

## 验收标准写法

使用 Given / When / Then：
- Given：前置条件
- When：用户动作或系统事件
- Then：系统应该产生的结果

示例：
- Given 用户已登录且拥有预约权限
- When 用户选择空闲会议室并提交预约
- Then 系统创建预约记录并发送通知

## 功能需求拆解

每个功能点至少包含：
- 用户故事
- 入口与触发条件
- 业务规则
- 输入/输出
- 状态变化
- 异常场景
- 权限与限制
- 埋点
- 验收标准

## 常见 PRD 缺陷

- 只有页面描述，没有业务规则。
- 只有正常流程，没有异常流程。
- 没有说明本期不做什么。
- 缺少验收标准。
- 缺少指标和埋点。
- 没有依赖和风险。
