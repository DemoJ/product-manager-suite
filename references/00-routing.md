# 任务路由与澄清策略

## 先判断用户真正要完成的产品工作

用户输入通常混合了问题、方案、情绪、指标和约束。先判断主要任务类型，再选择输出形式。

| 输入特征 | 任务类型 | 推荐输出 |
|---|---|---|
| “这个需求合不合理”“帮我 review PRD” | 需求评审 | 需求审查报告 |
| “帮我想想”“brainstorm”“还有什么方案” | 头脑风暴 | 头脑风暴报告 |
| “怎么做”“给个产品方案”“解决方案” | 方案设计 | 产品解决方案 |
| “写 PRD”“给研发看的文档” | PRD 写作 | 标准/轻量 PRD |
| “哪个先做”“排期”“roadmap” | 优先级/路线图 | 排序表 + 版本规划 |
| “怎么验证”“指标”“埋点”“AB” | 指标实验 | 实验设计/指标方案 |
| “怎么和老板/研发说”“拒绝需求” | 干系人沟通 | 沟通稿/决策记录 |
| “上线后效果不好/复盘” | 复盘迭代 | 复盘报告 |
| “画原型””线框图””wireframe””做个 demo””页面长什么样” | 原型设计 | HTML 原型 + 配套说明文档 |

## 澄清问题策略

不要为了完整而一次问太多。只问会改变判断和方案的关键问题。

优先级最高的澄清问题：
1. 业务目标是什么？提升转化、留存、收入、效率、满意度，还是降低成本？
2. 目标用户是谁？在哪个场景下遇到问题？
3. 当前用户如何解决？现有方案有什么不足？
4. 成功标准是什么？上线后看哪个指标？
5. 时间、人力、技术、合规、数据上有什么约束？

## 信息不足时的输出方式

当信息不足：
- 先说明“当前信息不足以定论”。
- 列出 3-5 个关键待确认问题。
- 同时基于现有信息给出初步判断和建议，不要只反问。

## 复杂任务拆解

如果用户请求很大，例如”帮我做一个完整产品方案”，按以下结构拆：
1. 问题定义
2. 用户与场景
3. 目标与指标
4. 方案发散
5. 推荐方案
6. MVP 范围
7. **原型设计**（将方案可视化，用于确认交互逻辑）
8. PRD/验收标准（基于确认的原型）
9. 风险与依赖
10. 下一步行动

**关键节点说明：**
- 原型设计在方案确认后、PRD 撰写前进行。
- 原型用于验证交互逻辑和用户体验，是 PRD 的重要输入。
- PRD 中的用户流程、交互说明需要和原型保持一致。
