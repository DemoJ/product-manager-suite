# 干系人沟通

## 沟通目标

产品经理的沟通不是转述需求，而是对齐目标、澄清取舍、推动决策。

## 向老板/业务方沟通

结构：
1. 我理解的业务目标
2. 当前问题与证据
3. 可选方案对比
4. 推荐方案与原因
5. 风险和需要决策的点
6. 下一步计划

当老板给出指定方案时，先尊重目标，再还原问题：
- “我理解您希望解决的是……”
- “这个方案可以解决其中一部分问题，但还有两个风险……”
- “我建议先用 MVP 验证……”

## 向研发沟通

研发更需要确定性和边界：
- 本期范围
- 非本期范围
- 核心流程
- 业务规则
- 异常场景
- 权限与状态
- 接口/数据依赖
- 验收标准

## 向设计沟通

设计更需要用户场景和体验目标：
- 用户是谁
- 用户当前路径
- 关键痛点
- 希望改善的体验指标
- 必须保留的业务规则
- 可探索的交互空间

## 拒绝或暂缓需求

拒绝时不要只说“不做”，要说明判断依据：
- 与当前目标不一致
- 用户价值证据不足
- 实现成本高于预期收益
- 存在更低成本替代方案
- 建议进入后续观察池

## 决策记录

重要会议后输出：
- 背景
- 已达成共识
- 待决策问题
- 负责人
- 截止时间
- 风险
- 下一步
